# Text & i18n

Canonical rules for every user-facing string across the `flowolf86` app suite
(`packliste`, `nebenkosten`, `scuba`, `gs`, `beikost`, `dashboard`). Source of truth is
`flowolf86/.github/standards/I18N.md`; each repo gets a synced copy at `.standards/I18N.md`
that its `CLAUDE.md` imports. **Edit the source, never the copies.**

The suite is bilingual (German + English, switched at runtime). The i18n *engine* is shared
(foundation-ui: `window.t(key)`, `window.tf(key, vars)`, the `data-i18n*` DOM pass, the FOUC
language bootstrap). This standard governs the *string catalogues* those consume.

## Architecture

- **Engine + shared chrome catalogue** live in **foundation-ui**. Chrome = strings identical
  across apps: auth/`login.*`, generic actions (`save`, `cancel`, `close`, `retry`, `undo`,
  `actions`, `delete`, `confirm`), the shared-shell `settings.*` controls (theme/language/
  motion), `nav.*`, `lang.*`, `error.404.*`. Change a shared string -> foundation-ui release +
  submodule bump. Do not copy chrome keys back into an app.
- **Each app owns only its domain catalogue**, one file per language under
  `app/static/locales/` (`de.js`, `en.js`). Each file is an IIFE that augments the global:
  ```js
  (function () {
    window.APP_TRANSLATIONS = window.APP_TRANSLATIONS || {};
    window.APP_TRANSLATIONS.de = { "nav.x": "...", /* ... */ };
  })();
  ```
  `i18n.js` merges these *over* the shell chrome catalogue -- **app keys win**, so an app
  overrides one chrome key by redefining just that key.
- **Never hardcode a user-facing string in Python.** Route it through the catalogue or keep it
  as localized data (`templates_data.py`-style bilingual records keyed by a stable ASCII slug).
  Never duplicate a catalogue key as a Python string literal.

## Catalogue file format

- **One file per language** under `app/static/locales/`: `de.js`, `en.js`. A German-only app
  ships only `de.js` (no empty `en.js`). The single-file `app/static/locales.js` (one file with
  `de:{}`/`en:{}` blocks) is **not** used -- convert any such app to the per-language files.
- Each file is the IIFE shown in Architecture, a flat map of `"<key>": "<value>"` pairs
  (double-quoted). Multiple pairs per line are fine; group with `// section` comments.
- Load them via a shared `app/templates/_locales.html` partial that `<script>`-includes every
  `locales/<lang>.js`, and point each page's `{% block app_translations %}` at
  `{% include "_locales.html" %}`.
- Keep the **same key order** across `de.js` and `en.js` so review diffs line up.

## Key grammar

- `^[a-z0-9]+(\.[a-z0-9]+)*$` -- lowercase ASCII, dot-separated segments.
- **No underscores** inside a segment. `reset.invalid_title` -> `reset.invalid.title`;
  `share.family.all_access` -> `share.family.allaccess`.
- **No `word+digit` numbering** to disambiguate (`list.rename.title2`, `copy.list.step1`). Use a
  role suffix (below) or split a genuine ordinal into its own numeric segment
  (`copy.list.step.1`). Pure-numeric segments are fine (`error.404.title`).

## Vocabulary (one word per concept)

Canonical trailing roles -- pick the exact word, no synonyms or abbreviations:

| Role | Use | Not |
|---|---|---|
| `.title` | heading of a page/dialog | -- |
| `.menu` | a menu-item / button label that opens the thing `.title` heads | `.title2` |
| `.subtitle` | secondary line under a title | `.sub`, `.subcaption` |
| `.desc` | longer descriptive body | `.body`, `.msg` (for descriptions) |
| `.label` / `.placeholder` / `.aria` / `.hint` / `.empty` | as named | -- |

- **One namespace per domain.** Do not run `template.*` and `templates.*` in parallel -- pick one.
- **Error keys, one shape:** `<domain>.error.<reason>` -- e.g. `login.error.credentials`,
  `save.error`, `templates.error.load`. Retire `*.failed`, `*.load.error`, `*.load.failed`.
- **Pluralization, one mechanism:** `window.tf()` with `<key>` (other/plural) + `<key>.one`
  (singular), selected by the count var. Do not encode plurals as free-form suffix words.

## Non-ASCII & the smart-quote escape rule

Literal UTF-8 is fine for letters and diacritics (a-umlaut, o-umlaut, u-umlaut, eszett),
dashes, ellipsis, middot, and emoji. **The five delimiter-confusable smart quotes must be
written as `\u` escapes** inside string values -- never as the literal character:

| Quote | Codepoint | Write it as |
|---|---|---|
| left single | U+2018 | `\u2018` |
| right single | U+2019 | `\u2019` |
| left double | U+201C | `\u201c` |
| right double | U+201D | `\u201d` |
| German low-9 (opening) | U+201E | `\u201e` |

So a German quote pair is written `\u201e{name}\u201c` and an English pair `\u201c{name}\u201d`.

**Why:** these glyphs are visually indistinguishable from ASCII `"`/`'` and survive editors, but
some tooling/API round-trips silently rewrite U+201D to ASCII `"`, which terminates the JS
string -> the catalogue parses fine locally, then ships a `SyntaxError` that breaks the whole UI
(this bit nebenkosten in production; see LESSONS). The `\uXXXX` form is pure ASCII, immune to the
round-trip, and evaluates to the identical character at runtime.

## de <-> en parity

- Every key present in `de` is present in `en` and vice versa.
- The set of `{placeholder}` tokens in a key's value is identical across `de` and `en`.
- **Monolingual catalogues are allowed.** An app may ship a single language (e.g. nebenkosten
  ships `de.js` only, by design). Parity is enforced only across the language files that are
  present, so a `de.js`-only app satisfies it trivially. Do not add a second language file just
  to satisfy the gate -- adding one is a translation decision, not a conformance fix.

## Enforcement

A shared gate -- `foundation.testing.locale_gate.make_locale_gate` -- enforces the mechanical
rules (grammar incl. numbering, de/en key + placeholder parity, the smart-quote charset rule).
Each app carries a one-line `app/tests/test_locales.py`:

```py
from foundation.testing.locale_gate import make_locale_gate

test_locale_naming, test_locale_parity, test_locale_charset = make_locale_gate(__file__)
```

It rides the existing `test`/`quality` pytest jobs (pure filesystem, no DB, no `node`), alongside
the `node --check app/static/*.js app/static/locales/*.js` syntax backstop. The gate logic needs a foundation release only
when these *rules* change; ordinary app text edits never do. The vocabulary rules above that
aren't mechanically checkable (word-per-concept, one-namespace-per-domain) are review-enforced.

**Interim:** until the shared gate ships in foundation-api-engine, each app carries a
self-contained copy of the gate logic in `app/tests/test_locales.py` (identical checks; mirrors
the existing per-app `test_overrides_documented.py`). The `foundation.testing` import above is the
target end-state — swapping the copy for the one-line import is a purely mechanical follow-up.
